# /ccm - Conventional Commit Manager

변경된 파일들을 분석하여 기능별로 분리된 커밋을 생성합니다.

## 실행 절차

### 1단계: 현재 상태 파악

다음 명령어를 순서대로 실행하여 변경 사항을 파악한다:

```bash
git status
git diff HEAD
git branch --show-current
```

### 2단계: 변경 사항 분석

- 변경된 파일들을 **기능 단위**로 그룹화한다
- 각 그룹에 적절한 커밋 태그를 결정한다

**커밋 태그 규칙:**
- `feat:` 새로운 기능 추가, 기존 기능의 요구 사항에 맞춘 수정
- `fix:` 기능에 대한 버그 수정
- `chore:` 패키지 매니저 수정, 기타 변경 사항 (예: `.gitignore` 수정)
- `docs:` 문서(주석) 수정
- `style:` 코드 스타일, 포맷팅에 대한 수정 (기능 변경 없음)
- `refactor:` 기능 변화 없이 코드 리팩터링 (예: 변수 이름 변경)
- `release:` 버전 릴리즈
- `merge:` 자신의 브랜치에 다른 브랜치를 병합할 때

### 3단계: 커밋 계획 수립

브랜치명에서 이슈 번호를 추출한다.
- 브랜치명이 `feature/123-something` 또는 `feat/123` 형식이면 `(#123)` 추출
- 브랜치명 자체가 숫자(`123`)면 `(#123)` 사용
- 그 외에는 브랜치명 전체를 `(#브랜치명)` 형식으로 사용

커밋 메시지 형식:
```
태그: 한국어로 간결하게 기능 설명 (#이슈번호)

- 제목만으로 표현이 부족한 중요한 내용이 있을 때만 하위 상세 설명 추가
```

### 4단계: 사용자 승인 요청

분석한 커밋 계획을 아래 형식으로 사용자에게 제시하고 **반드시 승인을 받은 후** 실행한다:

```
=== 커밋 계획 ===

[커밋 1]
태그: 설명 (#번호)

[커밋 2]
태그: 설명 (#번호)
- 제목으로 표현하기 어려운 주요 내용

이대로 커밋을 진행할까요? (수정이 필요하면 말씀해 주세요)
```

### 5단계: 커밋 실행

승인을 받으면 각 커밋을 순서대로 실행한다:

```bash
git add 파일1 파일2
git commit -m "태그: 설명 (#번호)"
```

멀티라인 커밋 메시지가 필요한 경우:
```bash
git commit -m "$(cat <<'EOF'
태그: 설명 (#번호)

- 상세 설명 1
- 상세 설명 2
EOF
)"
```

### 주의사항
- 절대로 사용자 승인 없이 커밋을 실행하지 않는다
- 커밋 메시지는 반드시 **한국어**로 작성한다
- 기능이 명확히 분리될 수 있으면 여러 커밋으로 나눈다
- `git add -A` 대신 파일을 명시적으로 지정한다
